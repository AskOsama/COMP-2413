Learning Objectives

By the end of this lecture, you will be able to:

  • Identify the three update anomalies (insert, update, delete) in a poorly designed table
  • Read and write functional dependencies using the X → Y notation
  • Distinguish super keys, candidate keys, primary keys, and prime / non-prime attributes
  • Decompose a table into First Normal Form (1NF) by removing repeating groups
  • Decompose a table into Second Normal Form (2NF) by removing partial dependencies
  • Decompose a table into Third Normal Form (3NF) by removing transitive dependencies
  • Recognize when BCNF is needed and how it differs from 3NF
  • Decide when controlled denormalization is justified

1. Why Normalize? Update Anomalies

The Problem: If a single table tries to store too many facts at once, every insert, update, and delete becomes risky. Normalization is the process of splitting tables so that each fact is stored in exactly one place.

Suppose a university used a single denormalized table to track which student takes which course and who teaches it:

Denormalized StudentCourse Table

Sid Sname Cid Cname Pid Pname Mark
1Mohamed Al TamimiC1Programming 23Dr. Ameera35
1Mohamed Al TamimiC3Programming 12Dr. Fahd22
2Sarra Al AskarC2Data Structure1Prof. Awartef17
2Sarra Al AskarC3Programming 12Dr. Fahd35
3Munera Al JomoaC2Data Structure1Prof. Awartef32

Three Anomalies This Causes

➕

Insert Anomaly

You can't add a brand-new course (e.g., a course nobody has taken yet) without inventing fake student data, because Sid is part of the row.

✏️

Update Anomaly

If "Dr. Fahd" changes name, you must update every row that mentions him. Miss one row and the database becomes inconsistent.

❌

Delete Anomaly

If the only student in course C1 drops it, deleting that row also wipes out the fact that course C1 exists and that Dr. Ameera teaches it.

Key Insight

All three anomalies have the same root cause: the table mixes independent facts (a student's name, a course's name, a professor's name, a mark) into one row. Normalization separates these facts into different tables.

2. Functional Dependencies

What is a Functional Dependency?

A functional dependency (FD) is a constraint between two sets of attributes in a relation. We write X → Y and read it as “X determines Y” or “Y is functionally dependent on X”.

It means: if two rows agree on the value of X, they must also agree on the value of Y.

Examples in the University Database

  • Sid → Sname, Age, Sex — the student ID determines the student's other attributes
  • Cid → Cname, Pid — the course ID determines the course name and which professor teaches it
  • Pid → Pname, Grade — the professor ID determines the professor's name and grade
  • Sid, Cid → Mark — the student/course pair determines the mark

Types of Functional Dependencies

Full Dependency

Y depends on the entire key, not just part of it. Example: {Sid, Cid} → Mark — you need both to know the mark.

Partial Dependency

Y depends on only part of a composite key. Example: in our table above, {Sid, Cid} → Sname is partial, because Sid → Sname already.

Transitive Dependency

X → Y and Y → Z, where Y is not a key. Example: Cid → Pid and Pid → Pname, so Cid → Pname is transitive.

Trivial Dependency

Y is a subset of X. Example: {Sid, Cid} → Sid. Trivially true; ignored when normalizing.

3. Keys Recap

Normal forms are defined in terms of keys, so let's nail down the vocabulary:

Term Definition Example (StudentCourse table)
Super key Any set of attributes that uniquely identifies a row {Sid, Cid}, {Sid, Cid, Mark}
Candidate key A minimal super key — remove any attribute and it stops being unique {Sid, Cid}
Primary key The candidate key chosen as the main row identifier {Sid, Cid}
Prime attribute An attribute that is part of some candidate key Sid, Cid
Non-prime attribute An attribute that is not part of any candidate key Sname, Cname, Pid, Pname, Mark

Why It Matters

2NF cares about partial dependencies on the primary key. 3NF cares about transitive dependencies between non-prime attributes. So you have to know which attributes are prime before you can apply the rules.

4. First Normal Form (1NF)

Rule

A relation is in First Normal Form if:

  1. Each cell holds a single, atomic value — no lists, no comma-separated values.
  2. There are no repeating groups of columns (e.g., Course1, Course2, Course3).
  3. Each column has a unique name and each row is uniquely identified by a primary key.

Bad: Not in 1NF

SidSnameCourses
1Mohamed Al TamimiC1, C3, C4
2Sarra Al AskarC2, C3

The Courses column packs many values into one cell. You can't easily ask “which students take C3?” without parsing the string.

Good: In 1NF

SidSnameCid
1Mohamed Al TamimiC1
1Mohamed Al TamimiC3
1Mohamed Al TamimiC4
2Sarra Al AskarC2
2Sarra Al AskarC3

Each cell now holds one value. The primary key becomes the composite {Sid, Cid}.

1NF is the baseline — if your table isn't 1NF, you can't even talk about 2NF or 3NF. SQL itself essentially assumes 1NF.

5. Second Normal Form (2NF)

Rule

A relation is in Second Normal Form if:

  1. It is already in 1NF.
  2. Every non-prime attribute is fully functionally dependent on the primary key — i.e., there are no partial dependencies.

2NF is only a concern when the primary key is composite. If the PK is a single column, partial dependencies cannot exist, so 1NF tables with single-column keys are automatically 2NF.

Bad: Not in 2NF

Take this 1NF table where the primary key is {Sid, Cid}:

Functional dependencies:

  • Sid → Sname  —  partial (depends on only part of the key)
  • Cid → Cname  —  partial
  • {Sid, Cid} → Mark  —  full

Good: Decomposed into 2NF

Split into three tables, each with a key that fully determines the rest:

6. Third Normal Form (3NF)

Rule

A relation is in Third Normal Form if:

  1. It is already in 2NF.
  2. No non-prime attribute is transitively dependent on the primary key.

Plain English: every non-prime attribute must depend on the key, the whole key, and nothing but the key. (A famous mnemonic attributed to a database designer named Bill Kent.)

Bad: Not in 3NF

Take this 2NF table for the Course catalog, where each course also stores its professor's name:

Cid (PK)CnamePidPnameGrade
C1Programming 23Dr. AmeeraAssistant Professor
C2Data Structure1Prof. AwartefProfessor
C3Programming 12Dr. FahdAssistant Professor
C4Statistic3Dr. AmeeraAssistant Professor

Functional dependencies:

  • Cid → Cname, Pid  —  full
  • Pid → Pname, Grade  —  non-prime determines non-prime  →  transitive

So Cid → Pname only because Cid → Pid and Pid → Pname. That's a transitive dependency — if Dr. Ameera changes name, two rows must be updated, and they could fall out of sync.

Good: Decomposed into 3NF

Course

Cid (PK)CnamePid (FK)
C1Programming 23
C2Data Structure1
C3Programming 12
C4Statistic3

Professor

Pid (PK)PnameGrade
1Prof. AwartefProfessor
2Dr. FahdAssistant Professor
3Dr. AmeeraAssistant Professor

Each professor's name now lives in exactly one row. To get a course's professor name, you join: Course ⋈ Professor ON Course.Pid = Professor.Pid.

Rule of Thumb

If a non-key column “describes” another non-key column, it probably belongs in its own table.

7. Boyce-Codd Normal Form (BCNF)

Rule

A relation is in BCNF if for every non-trivial functional dependency X → Y, the determinant X is a super key.

In other words: the only allowed determinants are super keys. BCNF is a slightly stricter version of 3NF.

Most tables that satisfy 3NF also satisfy BCNF. The two only differ in tables with overlapping candidate keys, which are uncommon in everyday designs.

When 3NF Is Not BCNF

Imagine a table StudentAdvisor(Sid, Major, AdvisorId) with rules:

  • Each student has one major and one advisor: {Sid, Major} → AdvisorId
  • Each advisor advises one major only: AdvisorId → Major

This is in 3NF (every non-prime attribute depends only on a candidate key), but AdvisorId is not a super key — yet AdvisorId → Major. So it's not BCNF.

Decomposition: StudentAdvisor(Sid, AdvisorId) + Advisor(AdvisorId, Major).

For most coursework and most real-world schemas, reaching 3NF is the practical goal. BCNF is a useful refinement when 3NF still leaves anomalies in tables with multiple overlapping candidate keys.

8. Worked Example: Step-by-Step Decomposition

Let's normalize the original denormalized table from Section 1, end to end.

Starting Point: Single Bloated Table

SidSnameCidCnamePidPnameMark
1Mohamed Al TamimiC1Programming 23Dr. Ameera35
1Mohamed Al TamimiC3Programming 12Dr. Fahd22
2Sarra Al AskarC2Data Structure1Prof. Awartef17

Step 1 — Reach 1NF

Each cell already holds one value. The composite primary key is {Sid, Cid}. Already in 1NF.

Step 2 — Reach 2NF

Identify partial dependencies on {Sid, Cid}:

  • Sid → Sname — partial → move to Student
  • Cid → Cname, Pid, Pname — partial → move to Course
  • {Sid, Cid} → Mark — full → stays in Marks

Tables after 2NF

Student(Sid, Sname)

Course(Cid, Cname, Pid, Pname)

Marks(Sid, Cid, Mark)

Step 3 — Reach 3NF

Check the Course table. Pid is non-prime and Pid → Pname — transitive. Pull professor info into its own table:

Final 3NF Schema

  • Student(Sid, Sname)
  • Professor(Pid, Pname)
  • Course(Cid, Cname, Pid→FK)
  • Marks(Sid, Cid, Mark) — Sid, Cid are also FKs

Sanity Check

This is exactly the schema used in Lectures 5 & 6+7. Those queries worked precisely because the schema was already in 3NF — no redundancy, clean joins.

9. A Note on Denormalization

Normalization improves integrity, but it splits data across many tables — meaning more joins at query time. For some read-heavy systems (analytics dashboards, reports), the cost of joining many tables on every query becomes a real performance concern.

Denormalization is the deliberate, controlled act of re-introducing some redundancy to speed up reads. It's a tradeoff:

AspectNormalizedDenormalized
StorageSmallerLarger (duplicated values)
WritesFast and simpleSlower, must update copies
ReadsNeed joinsOften single-table reads
IntegrityStrongRisk of inconsistency

Default to 3NF. Denormalize only when you have measured a real performance problem and you have a plan to keep the redundant copies in sync.

Practice Exercises

Apply What You've Learned

  1. Question 1: A library uses one table: Loan(MemberId, MemberName, BookId, BookTitle, AuthorId, AuthorName, LoanDate) with primary key {MemberId, BookId, LoanDate}. Which normal form does it violate, and how would you fix it?
    Show Answer

    It violates 2NF (partial dependencies: MemberId → MemberName, BookId → BookTitle, AuthorId, AuthorName) and also 3NF (transitive: BookId → AuthorId → AuthorName).

    Final 3NF schema:

    • Member(MemberId, MemberName)
    • Author(AuthorId, AuthorName)
    • Book(BookId, BookTitle, AuthorId→FK)
    • Loan(MemberId, BookId, LoanDate) — all three are FKs as well
  2. Question 2: Given the FDs A → B, B → C, and primary key A on table R(A, B, C), is R in 3NF? Explain.
    Show Answer

    No. A → B is fine, but B → C is a transitive dependency from the primary key (A) to C via B, where B is not a super key. Decompose to R1(A, B) and R2(B, C).

  3. Question 3: Why doesn't a relation with a single-column primary key need to be checked for 2NF specifically?
    Show Answer

    2NF only forbids partial dependencies on the primary key. Partial means “depends on part of a composite key”. A single-column key has no “part”, so partial dependencies are impossible. Any 1NF table with a single-column PK is automatically 2NF.

  4. Question 4: Convert this 1NF table to 3NF. Identify the FDs first.
    EmpIdEmpNameDeptIdDeptNameDeptManager
    E01AhmedD1SalesKhaled
    E02LaylaD2ITSami
    E03OmarD1SalesKhaled
    Show Answer

    FDs: EmpId → EmpName, DeptId; DeptId → DeptName, DeptManager. The PK is EmpId, so 2NF is automatic. But DeptId → DeptName is transitive → not in 3NF.

    3NF schema:

    • Department(DeptId, DeptName, DeptManager)
    • Employee(EmpId, EmpName, DeptId→FK)

Summary and Key Takeaways

What We've Learned

  1. Why Normalize: Denormalized tables suffer from insert, update, and delete anomalies because one row mixes independent facts.
  2. Functional Dependencies: X → Y means “X determines Y”. Full / partial / transitive dependencies drive the normal-form rules.
  3. 1NF: Atomic values, no repeating groups, primary key per row.
  4. 2NF: 1NF plus no partial dependencies on a composite primary key.
  5. 3NF: 2NF plus no transitive dependencies among non-prime attributes — the key, the whole key, and nothing but the key.
  6. BCNF: 3NF plus every determinant is a super key. Needed mainly when candidate keys overlap.
  7. Decomposition: The university schema (Student, Professor, Course, Marks) is already in 3NF — that's why the SQL queries in Lectures 5 and 6+7 stayed clean.
  8. Denormalization: A measured tradeoff for read-heavy workloads, not a default.
